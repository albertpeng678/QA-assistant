# PM 提案生成模板（通用 · 套用即生成）

> **這份文件本身就是一個 prompt。** 把你的 `spec.md` + `plan.md` 貼進下方 `<輸入文件>`，
> 連同整份模板一起丟給任何 LLM（Claude / GPT…），它會吐出 **7 頁可直接貼進 Gamma 的提案內容**。
>
> 適用任何產品題目 —— RAG agent 只是示範，換成你的題目照樣能用。

---

## 使用方式

1. 把你的 `spec.md` 與 `plan.md` 全文貼到下面 `<輸入文件>` 之間。
2. 把整份文件（含本說明、生成指令、7 個 STEP 區塊）一次丟給 LLM。
3. LLM 會逐一產出 STEP 1–7 的逐頁內容；抽不到的資訊會標 `【需補充：…】`。
4. 把產出貼進 Gamma → 生成簡報。

```
<輸入文件>
（把你的 spec.md 全文貼這裡）

---

（把你的 plan.md 全文貼這裡）
</輸入文件>
```

---

## 生成指令（給 LLM）

你是一位資深 PM 簡報教練。請**只根據 `<輸入文件>` 的內容**，為下方 7 個 STEP 逐一產出提案逐頁內容。規則：

- **主線**：指標貫穿全場 —— 先講產品靠什麼指標活、那指標生什麼病、你的功能怎麼治、最後用功能指標證明療效。7 頁要能串成這條故事線。
- 每個 STEP 先用一句話回答「核心問句」，再填「填空骨架」，最後整理成「輸出格式」的逐頁內容。
- **抽取規則**告訴你去 `<輸入文件>` 的哪裡找素材。依**三層**處理：
  1. **找得到** → 直接用。
  2. **找不到、但能依產品本質合理推斷**（常見於 STEP 1 北極星、STEP 3 使用者端、STEP 7 指標）→ **草擬一版**，並標 `【待確認：你推斷的依據】`，讓使用者確認或修改。
  3. **完全沒依據**（如真實訪談金句、實際流量數據）→ 標 `【需補充：要補什麼】`，**不要編造**。
- 每個 STEP 附了「104 範例」當**示範**（學它的具體程度與語氣，不要照抄它的內容）。
- 數字只在 STEP 7 與必要處出現；STEP 4 成功定義**不放數字**。
- 輸出語言：繁體中文。每頁輸出用 `## STEP N · 標題` 開頭，方便貼 Gamma。

## STEP 1 · 北極星（NORTH STAR）

- **核心問句**：這個產品靠什麼單一指標衡量「核心價值有沒有被兌現」？為什麼是它？
- **填空骨架**：
  - 北極星指標是：______（單一、跨團隊共享、領先而非落後）
  - 由這些要件構成：______（量的條件 + 質/投入的條件）
  - （選填）為什麼不是註冊數/營收：______
- **抽取規則**：到 spec 找「產品要解決的核心價值 / 目標 / 願景」段落；plan 若有指標/成功定義也一併看。把「價值主張」翻譯成一個可量測的領先指標。
- **104 few-shot 範例**：
  > 北極星 =「單月進站 ≥3 次 ＆ session 時長中位數 >10 分」的使用者數。
  > 為什麼：註冊數是漏斗早期、營收是落後指標；此定義同時抓回訪頻率（量）與投入深度（質）。
- **輸出格式**：
  ```
  ## STEP 1 · 北極星
  - 一句話：本產品的北極星是【指標】
  - 構成要件：量【…】＋ 質【…】
  - 為什麼是它（非註冊/營收）：【…】
  ```

## STEP 2 · 可優化處（BOTTLENECK）

- **核心問句**：北極星哪裡出問題？用什麼證據？為什麼值得解？
- **填空骨架**：
  - 觀察到的現象：______（用數據說話）
  - 背後的真正原因：______（把現象翻成「為什麼會這樣」）
  - 為什麼值得解：它導致 ______
- **抽取規則**：到 spec 找「問題定義 / Problem Statement / 痛點 / 現況不足」；plan 找促成此專案的動機。若 spec 無量化證據，標 `【需補充：佐證數據】`。
- **104 few-shot 範例**：
  > 現象：學測放榜時流量暴衝、其他時候很少人來（Similarweb，平日~1–2K vs 放榜日 33,765，落差約 20×）。
  > 真因：學生把它當「考試才用的工具」，不是「持續使用的好夥伴」。
  > 值得解：留不住人、淡季沒流量。
- **輸出格式**：
  ```
  ## STEP 2 · 可優化處
  - 現象（數據）：【…】
  - 真正原因：【…】
  - 為什麼值得解：【…】
  ```

## STEP 3 · 使用者端（USER NEEDS）

- **核心問句**：誰、在什麼情境、有什麼還沒被滿足的痛點？
- **填空骨架**：
  - Persona：______（聚焦單一角色）
  - 質性洞察 / 訪談金句：______
  - User Story：身為 ____，我想要 ____，以便 ____
  - 痛點收斂：______
- **抽取規則**：到 spec 找「目標使用者 / persona / user story / 使用情境」；plan 若有使用者場景一併看。沒有訪談金句就標 `【需補充：訪談洞察】`。
- **104 few-shot 範例**：
  > Persona：準備個人申請面試的高中職學生（訪談對象 Eric）。
  > 洞察：練習資源「有量無質」或「有質無量」。
  > User Story：身為高三考生，我想反覆練習面試並得到即時回饋，以便進考場前更有把握。
  > 痛點：缺乏兼具數量與品質的練習機會。
- **輸出格式**：
  ```
  ## STEP 3 · 使用者端
  - Persona：【…】
  - 洞察 / 金句：【…】
  - User Story：身為【…】，我想要【…】，以便【…】
  - 痛點：【…】
  ```

## STEP 4 · 成功定義（SUCCESS DEFINITION，這頁不放數字）

- **核心問句**：相對現況，什麼問題若被解決，提案就算成功？
- **填空骨架**：
  - 現況：______
  - 成功定義：提案要成功，必須讓 ______（質性、目標導向，**不放數字**）
- **抽取規則**：呼應 STEP 2 的問題，從 spec 的目標/驗收標準提煉一句**質性**的方向句。把任何數字移到 STEP 7。
- **104 few-shot 範例**：
  > 現況：面試練習資源「有量無質」或「有質無量」。
  > 成功定義：必須讓學生取得「兼具數量與品質」的練習機會。
- **輸出格式**：
  ```
  ## STEP 4 · 成功定義
  - 現況：【…】
  - 成功定義（不放數字）：必須讓【…】
  ```

## STEP 5 · 機制（MECHANISM）

- **核心問句**：功能透過什麼機制，推動北極星的哪一個構成要件？怎麼歸因？
- **填空骨架**：
  - 功能一句話：______
  - 作用機制：因為這功能具有 ______（需求性質），所以帶動 ______（北極星某分項）
- **抽取規則**：到 spec/plan 找「核心功能 / 解法」，把它的性質（重複性、深度、即時性…）連到 STEP 1 北極星的某個構成要件，講出因果。
- **104 few-shot 範例**：
  > 功能：104 AI 升學面試顧問。
  > 機制 1：非一次性需求（重複練習）→ 把單次造訪轉成多次 → 推升「月進站 ≥3 次」。
  > 機制 2：深度互動屬性 → 帶動 session 時長 → 直接貢獻「session 時長 >10 分」。
- **輸出格式**：
  ```
  ## STEP 5 · 機制
  - 功能一句話：【…】
  - 機制 →北極星分項：因為【性質】，所以帶動【北極星分項】
  ```

## STEP 6 · 解方展現（SOLUTION）

- **核心問句**：這功能長什麼樣？關鍵流程 / 畫面為何？
- **填空骨架**：
  - 核心流程：______（主要步驟）
  - 關鍵畫面：______（wireframe / 示意）
  - （選填）參考競品與差異：______
  - （選填）技術可行性重點：______
- **抽取規則**：到 plan 找「實作步驟 / 架構 / 流程 / 畫面」；spec 找競品或技術約束。流程用「步驟 1→2→3」表達。畫面可引用你做的 mockup。
- **104 few-shot 範例**：
  > 流程：登入 → 開始面試 → 逐題作答 → AI 回饋 → 結束頁雷達圖。
  > 競品：Google Interview Warmup、PrepMeUp、智面星。
  > 技術：固定題庫可不走 AOAI；STT 用較便宜的 Whisper。
- **輸出格式**：
  ```
  ## STEP 6 · 解方展現
  - 核心流程：【步驟1】→【步驟2】→…
  - 關鍵畫面：【描述 / 連結 mockup】
  - 競品差異：【…】
  - 技術重點：【…】
  ```

## STEP 7 · 驗證計畫（VALIDATION，兩層指標 + 護欄）

- **核心問句**：單看這功能怎樣算成功？它在指標體系裡的位置？有哪些護欄？
- **填空骨架**：
  - 北極星（產品線層）：______（同 STEP 1）
  - 功能層成功指標：______（布林條件 · 量+質代理 · 含時長上下限）
  - 量能指標：______（有沒有人用、用到底）
  - 護欄指標：______（防灌量/灌時長的假成功）
- **抽取規則**：北極星同 STEP 1。功能層指標要可量測、含品質代理。量能=使用量；護欄=防止刷量造假。若 spec/plan 無指標，標 `【需補充：量化指標】`。
- **104 few-shot 範例**：
  > 北極星：「單月進站 ≥3 次 ＆ session 時長中位數 >10 分」的使用者數。
  > 功能層成功指標：單月完成 ≥2 次、單次時長 10–30 分 的使用者數。
  > 量能：練習啟動數、練習完成數。
  > 護欄：平均練習時長（排除極端值）不下滑、單次平均完成題數不下滑。
- **輸出格式**：
  ```
  ## STEP 7 · 驗證計畫
  - 北極星：【同 STEP 1】
  - 功能層成功指標：【…】
  - 量能指標：【…】
  - 護欄指標：【…】
  ```

---

## 收尾（給 LLM）

7 頁產出後，附一段 **3–5 句的口頭摘要**，把這 7 頁串成「指標貫穿全場」的一段話，方便面試/提案開場時講。並列出所有 `【需補充】` 清單，提醒使用者補齊。
