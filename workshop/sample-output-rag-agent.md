## STEP 1 · 北極星
- 一句話：本產品的北極星是【單月「得到帶正確引用之答案」的有效問答使用者數】
- 構成要件：量【單月發出 ≥3 次查詢】＋ 質【回答附正確法條引用、使用者未重問同題】
- 為什麼是它（非查詢總數/註冊數）：查詢總數含大量無效/亂試；註冊是漏斗早期。此定義同抓「回來用」（量）與「答得準」（質）。
- 【待確認：spec 為教學參考、未定義北極星；此版依「法規問答工具的核心價值＝準確且可信地回答」推斷，請確認指標與門檻】

## STEP 2 · 可優化處
- 現象（數據）：以 file search 預設 chunking（800/400）索引法規，法條檢索命中破碎、引用不精準。【需補充：實測檢索準確率數據】
- 真正原因：法規是高度結構化文本（條文層級、跨條引用），通用 chunking 切斷條文邊界，使檢索片段失去語意完整性。
- 為什麼值得解：引用不準 → 使用者不信任答案 → 法規場景對「可追溯到第幾條」要求極高，不準等於不可用。

## STEP 3 · 使用者端
- Persona：需要快速查特定法規條文的【非法律專業工作者】（如 HR 查勞基法、會計查稅法）。
- 洞察：他們不要長篇法律解釋，要「這件事依第幾條、條文怎麼說」。【需補充：實際訪談金句】
- User Story：身為【需處理法遵事務的承辦】，我想要【用白話問法規並得到附條號的答案】，以便【不必逐條翻法典就能確認依據】。
- 痛點：自己 Google 法條易過時/斷章取義，逐條翻法典太慢。
- 【待確認：spec 未含 persona；此版依法規問答典型使用者推斷，請確認對象是否為 HR/會計/一般民眾】

## STEP 4 · 成功定義
- 現況：查法規要嘛 Google 到不可靠來源、要嘛逐條翻法典。
- 成功定義（不放數字）：必須讓使用者用白話提問，就能得到**帶正確條文引用、可追溯**的答案。

## STEP 5 · 機制
- 功能一句話：以 OpenAI file search 對法規語料做檢索增強問答，回答附條號引用。
- 機制 →北極星分項：因為【回答附可驗證的條文引用】（可信性質），所以帶動【「答得準、不重問」這個質的分項】；因為【白話即可查、降低查詢門檻】，所以帶動【月查詢 ≥3 次 這個量的分項】。

## STEP 6 · 解方展現
- 核心流程：使用者輸入問題 →【/api/ask】→ OpenAI Responses API（file_search, vector_store）→ 取答案 + annotations →【前端顯示答案與引用來源】。
- 關鍵畫面：單頁聊天介面，答案下方以卡片列出引用法條檔名/條號（見 mockup）。
- 競品差異：相較通用 ChatGPT，本方案綁定**受控法規語料 + 強制引用**，可追溯、不幻想條文。
- 技術重點：單一 FastAPI 服務 + 靜態前端；chunking 用 static 512/128（經 context7 研究法規 RAG 主流作法對齊）；GitHub 連結式 CICD 部署 Railway。

## STEP 7 · 驗證計畫
- 北極星：單月「得到帶正確引用之答案」的有效問答使用者數（同 STEP 1）。
- 功能層成功指標：單月 ≥3 次查詢、且「答案含引用」比例 ≥ X%、「同題重問率」≤ Y% 的使用者數。【待確認：X/Y 門檻待定】
- 量能指標：查詢發起數、完成（拿到答案）數。
- 護欄指標：引用正確率不下滑（防為了有引用而亂引）、平均回答延遲不上升（防體驗劣化）。
- 【待確認：spec 僅提 Sentry 監控、未定義指標體系；此版依北極星反推功能層/量能/護欄，請確認門檻】

## 口頭摘要（開場 30 秒）
法規問答的核心價值是「答得準又可追溯」。現況用通用 chunking 索引法規，檢索破碎、引用不準，使用者不敢信。我們用 file search + 強制條文引用，並針對法規結構調 chunking；機制上，可信引用推動「答得準」、白話查詢推動「常回來用」，共同抬升北極星——單月得到可信引用答案的有效使用者數。驗證採兩層指標，並用引用正確率與延遲當護欄，擋住「為有引用而亂引」的假成功。
